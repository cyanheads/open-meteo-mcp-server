# 0.2.0 — 2026-07-02

## Added

- **`openmeteo_get_climate`** — bias-corrected daily CMIP6 climate projections from the Open-Meteo Climate API (`/v1/climate`), covering 1950-01-01 to 2050-12-31 at any coordinate — the future-projection counterpart to `openmeteo_get_historical`. Selectable models: `CMCC_CM2_VHR4`, `FGOALS_f3_H`, `HiRAM_SIT_HR`, `MRI_AGCM3_2_S`, `EC_Earth3P_HR`, `MPI_ESM1_2_XR`, `NICAM16_8S`; with 2+ models each variable column is suffixed with the model name (e.g. `temperature_2m_max_CMCC_CM2_VHR4`), and model/variable gaps return null. Validates date order and the 1950–2050 coverage window server-side; upstream invalid variable/model rejections are reframed via `frameInvalidVariableMessage`. Results past 500 records spill to DataCanvas under the same pattern as the other producers (`canvas_id` present only when `truncated: true`). ([#2](https://github.com/cyanheads/open-meteo-mcp-server/issues/2))
- **`getClimate()` / `ClimateParams`** on `OpenMeteoService`, with a new `climateBaseUrl` config field — override via the optional `OPEN_METEO_CLIMATE_BASE_URL` env var (default `https://climate-api.open-meteo.com`).

## Changed

- **Dataframe tools name three canvas producers.** `openmeteo_dataframe_describe` / `openmeteo_dataframe_query` descriptions, `canvas_id` input docs, and `canvas_not_found` recovery hints now list `openmeteo_get_climate` alongside `openmeteo_get_historical` and `openmeteo_get_ensemble`; server instructions, `CANVAS_PROVIDER_TYPE` docs (`server.json`, `manifest.json`, `.env.example`), and README follow.
- **README config table** now documents the `OPEN_METEO_ENSEMBLE_BASE_URL` and `OPEN_METEO_FLOOD_BASE_URL` overrides, which were missing rows.
