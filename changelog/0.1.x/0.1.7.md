# 0.1.7 — 2026-07-02

## Changed

- **Dependencies:**
  - `@cyanheads/mcp-ts-core` `^0.10.9` → `^0.10.10`
  - `@biomejs/biome` `^2.5.0` → `^2.5.2`
  - `@types/node` `^26.0.0` → `^26.1.0`

## Fixed

- **`openmeteo_geocode` — sparse results no longer hard-fail.** The Geocoding API omits `country`/`country_code` for non-country features (continents, oceans — e.g. "Antarctica", `feature_code: "CONT"`) and can omit `timezone`; the output schema now declares those fields nullable, the handler coalesces them to `null`, and `format()` omits missing markers instead of rendering `null`. ([#5](https://github.com/cyanheads/open-meteo-mcp-server/issues/5))
- **`openmeteo_geocode` — native-script queries resolve.** A query in a recognized non-Latin script (Kana → `ja`, Hangul → `ko`, Han → `zh`, Cyrillic → `ru`, Arabic, Greek, Hebrew, Thai, Devanagari) that misses under the default `language: "en"` is retried once with the language inferred from its script — `{ name: "上海" }` now returns Shanghai instead of `no_results`. The `language` description and the `no_results` recovery hint now tell callers to match `language` to the script of `name` (the old hint said "search in English"). ([#10](https://github.com/cyanheads/open-meteo-mcp-server/issues/10))
- **`openmeteo_get_ensemble` — `model` and `member_count` populated.** Both output fields were sourced from top-level `models`/`members` envelope fields the Ensemble API never returns, so they were always absent. `model` now echoes the requested `models` parameter (absent for the API default blend), and `member_count` is counted from distinct `_memberNN` column suffixes across the hourly/daily blocks. ([#6](https://github.com/cyanheads/open-meteo-mcp-server/issues/6))

## Security
